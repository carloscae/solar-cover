# Solar Cover — Design Spec

**Date:** 2026-05-27
**Status:** Approved

---

## Problem

Adaptive-cover is the dominant HA cover automation integration on HACS. It has three fundamental problems:

1. **Breaks in winter** — the horizontal awning formula divides by `sin(solar_elevation)`. At low winter elevations this produces extension values of 200–900%, clamped to 100% (fully deployed), blocking the passive heating the user actually wants. The clip that was meant to prevent this is commented out in the source.
2. **56 parameters with non-obvious interactions** — most encode intent indirectly through geometric constraints, creating a system that is hard to configure and impossible to debug.
3. **No observability** — the integration cannot tell the user why it made a decision.

---

## Goal

A Home Assistant custom integration that controls solar covers (vertical blinds, horizontal awnings, tilt/venetian blinds) correctly year-round, with a configuration surface that maps to how users actually think about their setup, and attributes that explain every decision.

Secondary goal: publishable to HACS, feeds into OSS builder-authority strategy.

---

## Non-goals (v1)

- Lovelace companion card (backlog — position curve attribute is ready for it)
- Machine learning / adaptive correction from manual overrides
- Active hours / time window restriction (handle with HA automation)
- Blind spot configuration

---

## Architecture

Two configuration tiers:

```
Integration  →  Cover Zone(s)
(global)        (one per group of covers that move together)
```

No facade abstraction. A Cover Zone is one config entry that controls one or more physical cover entities. Covers that move identically share a zone; covers with different geometry get separate zones.

## Configuration

### Integration (global, configured once)

| Parameter | Type | Notes |
|---|---|---|
| Weather entity | Entity (optional) | Gates on precipitation + wind |
| Wind speed threshold | Float (optional) | m/s — used if weather entity has wind data |
| Min outside temperature | Float (optional) | °C — don't deploy below this |
| Default inactive position | Int | Where covers go when not shading. Default: 0% (retracted) |
| Default manual override duration | Int | Minutes to pause after user manual move. Default: 120 |

| Parameter | Type | Notes |
|---|---|---|
| Name | String | e.g. "South Terrace" |
| Cover entities | Entity list | Multi-select, all receive same position |
| Cover type | Enum | vertical / horizontal / tilt |
| Azimuth | Int (°) | Compass bearing of the opening |
| FOV left | Int (°) | 1–90 for vertical/tilt, 1–180 for horizontal |
| FOV right | Int (°) | 1–90 for vertical/tilt, 1–180 for horizontal |
| Elevation threshold | Float (°) | Auto-computed from latitude: `(90 - lat) * 0.6`. Adjustable. |
| Elevation threshold override | Float (optional) | Per-zone override of integration default |
| Inactive position override | Int (optional) | Per-zone override of integration default |

| Parameter | Notes |
|---|---|

| Glare protection depth (m) | How far from the window sun should not reach |

**Horizontal awning geometry:**
| Parameter | Notes |
|---|---|
| Attachment height above floor (m) | Height where awning is mounted |
| Awning span length (m) | Physical maximum extension |

| Parameter | Notes |
|---|---|
| Slat width (mm) | |
| Slat spacing (mm) | Must be ≤ slat width — validated in config flow |

| Parameter | Default | Notes |
|---|---|---|

**Total: ~17–21 parameters per zone depending on cover type. All load-bearing.**

## Solar Engine (`solar.py`)

- Reads `hass.config.latitude`, `hass.config.longitude`, `hass.config.elevation` — zero user config
- Uses `astral` (already a HA core dependency — no new package declared)
- No runtime dependency on `sun.sun` entity

Key methods:

```python
def sun_position(dt: datetime) -> tuple[float, float]:
    """Return (azimuth, elevation) for any datetime."""

def position_curve(date: date, interval_minutes: int = 5) -> list[SunSample]:
    """288 samples for a given date. Cached; refreshes at midnight."""

def fov_window(azimuth: int, fov_left: int, fov_right: int) -> tuple[datetime, datetime]:
    """Today's FOV entry and exit times."""
```

The position curve is returned as a list of `(time, azimuth, elevation, position_pct)` samples. This attribute is attached to each cover entity, ready for a future Lovelace card without any additional work.

## Intent Model (`intent.py`)

Sequential gate — evaluation stops at the first failing gate:

```
1. Sun above elevation threshold?
       No  → intent: INACTIVE_SUN_LOW
       Yes ↓

2. Sun within FOV (gamma within fov_left / fov_right)?
       No  → intent: INACTIVE_OUTSIDE_FOV
       Yes ↓

3. Weather conditions acceptable?
   (no precipitation, wind below threshold, temp above min)
       No  → intent: INACTIVE_WEATHER
       Yes ↓

4. Manual override active?
       Yes → intent: MANUAL_OVERRIDE
       No  ↓

   → intent: SHADING
   → compute position via geometry engine
```

The `intent` value is exposed as an entity attribute. "Why is my awning retracted?" is always answerable.

## Geometry Engine (`geometry.py`)

Pure functions — no side effects, fully unit-testable independently of HA.

**Vertical blind:**
```python
def vertical_position(sol_elev, gamma, distance, h_win) -> float:
    """Returns blind height needed to block sun at glare depth."""
    blind_height = clip((distance / cos(gamma)) * tan(sol_elev), 0, h_win)
    return blind_height / h_win * 100
```

**Horizontal awning:**
```python
def horizontal_position(sol_elev, gamma, h_win, awn_length, awn_angle, distance) -> float:
    """Returns awning extension % needed to protect glare depth."""
    # Base vertical step — when gamma > 90°, cos is negative → blind_height clips to 0
    # → full extension (100%). Correct: oblique sun needs maximum shade.
    blind_height = clip((distance / cos(gamma)) * tan(sol_elev), 0, h_win)
    a_angle = 90 - sol_elev
    c_angle = sol_elev + awn_angle
    length = ((h_win - blind_height) * sin(a_angle)) / sin(c_angle)
    return clip(length / awn_length * 100, 0, 100)  # clip restored — was commented out in adaptive-cover
```

**Tilt blind:**
```python
def tilt_position(sol_elev, gamma, slat_distance, slat_depth, bidirectional) -> float:
    """MDPI formula with NaN guard when discriminant goes negative."""
    beta = arctan(tan(sol_elev) / cos(gamma))
    discriminant = tan(beta)**2 - (slat_distance / slat_depth)**2 + 1
    if discriminant < 0:
        return 100.0  # fully closed — sun too low for slat geometry to help
    slat = 2 * arctan((tan(beta) + sqrt(discriminant)) / (1 + slat_distance / slat_depth))
    pct = degrees(slat) / (180 if bidirectional else 90) * 100
    return clip(pct, 0, 100)
```

## Coordinator

- One coordinator per Cover Zone
- **Update trigger:** time-based (every 5 minutes) + `state_changed` on weather entity
- No `sun.sun` listener — internal calculation, internal schedule
- On update:
  1. Compute current sun position
  2. Run intent model
  3. If `SHADING`: compute position via geometry engine
  4. Apply min/max clamp
  5. If position delta > hysteresis threshold: command all zone entities
  6. Update entity attributes

## Observability

Every cover entity exposes these attributes:

| Attribute | Example value |
|---|---|
| `intent` | `shading` |
| `sun_azimuth` | `234.6` |
| `sun_elevation` | `40.2` |
| `surface_azimuth` | `-67.6` (gamma) |
| `computed_position` | `72` |
| `commanded_position` | `72` (may differ from computed if clamped) |
| `fov_entry` | `07:04` |
| `fov_exit` | `17:15` |
| `position_curve` | `[{"time": "06:00", "pct": 0}, ...]` (24 hourly points for UI; internal calculation uses 288 five-minute samples for precise FOV window computation) |

`intent` is the primary debugging surface. When something unexpected happens, this attribute explains why.

## Config Flow UX

**Step 1 — Integration setup (once)**
Weather entity picker (optional), wind threshold, min temperature, default inactive position, default override duration. All optional fields skippable.

**Step 2 — Zone setup**
- Name the zone
- Entity multi-select (filtered to covers with SET_POSITION support)
- Cover type selector
- Azimuth input
- FOV left/right sliders (range adjusts based on cover type)
- Elevation threshold: shown with auto-computed value and plain-language label: "Covers activate when the sun is above this angle — lower values mean more shading in spring and autumn."
- Geometry inputs: labelled per selected cover type, with unit hints and physical descriptions

**Auto-suggest on azimuth entry:** if azimuth is within ±10° of an existing zone, surface: "This looks like [Zone Name] (167°) — copy those geometric settings?" One click to accept.

## Tech Stack

| Choice | Rationale |
|---|---|
| Python 3.12+ | HA minimum for new integrations |
| `astral` | Already HA core dependency — no new package |
| `numpy` | Already HA core dependency — geometry math |
| No `pandas` | Overkill for a list of 288 floats — plain list comprehension |
| Keyword arguments throughout | No positional dataclass unpacking — adaptive-cover's silent failure mode |
| `pytest` + `pytest-homeassistant-custom-component` | Test suite from day one |
| Pure functions for geometry | Testable without HA running |

## Known Differences from Adaptive-Cover

| Adaptive-Cover | Solar Cover |
|---|---|
| 56 parameters | 17–21 depending on type |
| 3 config entry types (global + facade + cover) | 2 (integration + zone) |
| One entity per motor, separate config | Multiple entities per zone |
| Climate mode drives position calculation | Intent model gates deployment only |
| sun.sun entity dependency | Internal astral calculation |
| pandas for 289-point daily curve | Plain list, numpy for geometry |
| FOV capped at 90° for all types | Horizontal allows up to 180° |
| `calculate_position()` clip commented out | Clip restored, NaN guard added for tilt |
| Zero tests | pytest suite from day one |
| No explanation for position decisions | `intent` attribute on every entity |
